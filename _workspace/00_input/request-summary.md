# Request Summary — Visual/audio Phase 7.3 adaptive music stems v0.12.72

## Authorized outcome

Complete Milestone 7.3 as a bounded, fixture-verifiable adaptive music stem
system. Keep music optional, visible-state-only, context/intensity-oriented,
and outside deterministic host/simulation state.

## Slice boundary

- Cover menu/planning, stable operations, pressure, regulatory scrutiny,
  competitive escalation, affiliation/negotiation, and debrief.
- Define five stem roles per state: base pulse, institutional motif, visible
  pressure layer, policy layer, and transition cadence, with visible trigger,
  bounded crossfade, source hash, text equivalent, and fallback metadata.
- Extend the runtime catalog and visible classifier while preserving the
  existing four state IDs and deterministic replay-state sequence planning.
- Add music-only mute and test music-only/full mute plus cues-only behavior.
- Do not add recorded audio, hidden-state selection, Phase 7.4 priority/fatigue
  management, or simulation/host/replay changes.

## Validation target

Focused music/audio tests, fixture proof checks, registry/credits, metadata,
documentation links, full Python, full Rust, formatting, and diff checks.

## Evidence limits

Generated stem metadata, deterministic source hashes, visible-state classifier
tests, and replay planning establish technical coverage only. They do not
establish audibility, musical quality, fatigue, lived accessibility, classroom
suitability, human comprehension, learning, calibration, or policy validity.
