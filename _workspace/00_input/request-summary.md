# Request Summary — Visual and Audio Phase 1 Static Desktop v0.12.17

## Scope

- Implement the proposal's Phase 1 static executive desktop over representative
  injected actor-visible data without claiming that live presentation features
  already exist.
- Surface executive metrics, briefing, regional schematic, systems/facilities,
  selected detail, contextual action previews, pending processes, monthly result,
  history, and debrief in a responsive semantic layout.
- Preserve the Phase 0 browser-native/adapter boundary, promote only Phase 1,
  record Phase 2 as the next gate, and align version `0.12.17` metadata.

## Sources

- `docs/visual_audio_upgrade_proposal.md`, Phase 0 alignment, and ADR-0011.
- Current Rust, MCP, GUI proof, scenario, replay, and debrief surfaces.
- Canonical proposal, roadmap, design principles, harness team spec, SPEC,
  architecture, ADRs, changelog, lessons, and versioning policy.

## Expected files

- Phase 1 static desktop, experience/checklist document, and contract test.
- `SPEC.md`, `ARCHITECTURE.md`, `CHANGELOG.md`, and release metadata.
- Evidence/mechanism/domain-QA handoffs, lessons, and durable final handoff.

## Validation target

- Phase 1 surface, user-task, hidden-state, responsive-layout, and asset/network
  checks.
- Release metadata, documentation, and full Rust verification.

## Explicit non-goals

No simulation, balance, scenario, typed DTO, live adapter, audio playback, asset
acquisition, packaging, publication, deployment, or calibration changes. Static
checks do not establish human usability, engagement, accessibility lived
experience, learning, classroom effectiveness, domain-expert validity, or policy
validity.

## Global workflow

Use the repo orchestrator and end-user experience workflow for Phase 1
presentation work, plus the global simple-code, spec-driven, and preferred
workflow skills. No Rust implementation workflow is required unless the audit
uncovers a code defect.
